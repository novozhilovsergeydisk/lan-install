## [ ] 1. Импорт: разбор адресов с «район/поселок» + чистка существующих

**Проблема.** `App\Services\PlanningRequest\ExcelRequestParser::parseAddressString` умеет только
формат «город, улица, дом». Для «город Москва, **район Внуково**, улица Анны Ахматовой, д. 18»
он кладёт всё после города в `street` (= «район Внуково, улица Анны Ахматовой»), а `district`
оставляет пустым. Срез ведущего «улица» (`AddressMatcher::stripLeadingStreetType`) тут не помогает —
ведущее слово «район». Такие записи к тому же — скрытые дубли «чистых» адресов (напр. «район Внуково,
улица Анны Ахматовой / 18» дублирует уже существующий «Анны Ахматовой / 18, посёлок Внуково»),
которые `addresses:merge-duplicates` не склеивает (сравнивает точный текст улицы).

**Важно не сломать.** Легитимные адреса Новой Москвы, где «посёлок X» / «микрорайон X» — само место
(улицы нет), и `district` заполнен. Их трогать нельзя. Чистить только записи с ведущим «район» и
**пустым** `district`, по списку id, а НЕ по маске.

**Чеклист:**
- [ ] Собрать реальные форматы адресов из исходных Excel заказчика (как пишут район/поселок/поселение/микрорайон, порядок сегментов).
- [ ] Доработать `parseAddressString`: распознавать ведущий «район …[, поселок …]», выносить район в `district`, оставлять настоящую улицу в `street`. Определить правило для New-Moscow без улицы.
- [ ] Юнит-тесты парсера на новые форматы (включая адреса без улицы).
- [ ] Команда/скрипт точечной переразборки битых записей (dry-run по умолчанию). На 2026-06-16 их было 6: `#2153, #2154, #2160, #2162, #2171, #2176` — **перепроверить запросом**, id могли измениться.
- [ ] `php artisan addresses:merge-duplicates` — схлопнуть всплывшие дубли с «чистыми», проверить перепривязку заявок.
- [ ] `php artisan addresses:geocode-missing` для переразобранных (после включения «Стандартизации» у токена DaData).
- [ ] Локально на копии прод-БД → проверка → деплой → на проде выполнить чистку/merge в порядке.

## [x] 3. Оборудование под бригадой: «живой» режим без лага (рефреш при загрузке)

**Реализовано 2026-06-22** (локально, тесты 10/10, проверено Playwright) — ждёт ручной проверки и деплоя.

**Контекст.** Оборудование (комплект H-* + машина) показывается рядом с бригадой для ОТКРЫТЫХ
заявок только за сегодня; данные в `request_equipment` наполнялись командой `wms:refresh-equipment`
(`->hourly()`), при закрытии — заморозка. Минус был: лаг до часа между выдачей и появлением.

**Важно (выяснено при отладке):** дневную таблицу на загрузке перерисовывает `handler.js`
(fetch `/api/requests/date/{today}` = `getRequestsByDate`) и сам рендерит оборудование из
`request.equipment`. Поэтому JS-инъекция своего блока бесполезна (перетирается). Правильное
решение — делать **данные свежими на сервере**, и тогда и серверный рендер, и handler.js покажут свежее.

**Сделано (серверный подход):**
- [x] `WmsEquipmentService::refreshTodayBestEffort()` — обновляет `request_equipment` открытых
      сегодняшних заявок из склада. Вызывается в `HomeController::index()` и `getRequestsByDate()` (за сегодня),
      перед отдачей данных. Так при перезагрузке инвентарь свежий сразу после выдачи, а не раз в час.
- [x] Троттл 30 сек (`Cache` `wms_equipment_refreshed_at`) — частые F5 склад не долбят.
- [x] Безопасность главной: быстрый пинг склада (2 сек) — недоступен → выходим, страницу не вешаем; всё в try/catch.
- [x] Cron `wms:refresh-equipment` оставлен `->hourly()` как «пол».
- [x] Прод-инфра: планировщик Laravel в crontab настроен 2026-06-22 (`sudo -u www-data … schedule:run`), см [[prod-scheduler-www-data]].
- [x] Проверено Playwright: протухший маркер на перезагрузке перезаписывается реальными данными склада;
      при активном троттле — рефреш пропускается (склад не дёргается).
- [ ] (Опц., НЕ делали) «по-настоящему живое» без F5 (поллинг/пуш) + батч-эндпоинт на складе —
      только если per-load опрос станет узким местом.
