## [ ] 1. Импорт: разбор адресов с «район/поселок» + чистка существующих

**Проблема.** `App\Services\PlanningRequest\ExcelRequestParser::parseAddressString` умеет только
формат «город, улица, дом». Для «город Москва, **район Внуково**, улица Анны Ахматовой, д. 18»
он кладёт всё после города в `street` (= «район Внуково, улица Анны Ахматовой»), а `district`
оставляет пустым. Срез ведущего «улица» (`AddressMatcher::stripLeadingStreetType`) тут не помогает —
ведущее слово «район». Такие записи к тому же — скрытые дубли «чистых» адресов (напр. «район Внуково,
улица Анны Ахматовой / 18» дублирует уже существующий «Анны Ахматовой / 18, посёлок Внуково»),
которые `addresses:merge-duplicates` не склеивает (сравнивает точный текст улицы).

**Важно не сломать.** Легитимные адреса Новой Москвы, где «посёлок X» / «микрорайон X» — само место
(улицы нет), и `district` заполнен. Их трогать нельзя. Чистить только записи с ведущим «район» и
**пустым** `district`, по списку id, а НЕ по маске.

**Чеклист:**
- [ ] Собрать реальные форматы адресов из исходных Excel заказчика (как пишут район/поселок/поселение/микрорайон, порядок сегментов).
- [ ] Доработать `parseAddressString`: распознавать ведущий «район …[, поселок …]», выносить район в `district`, оставлять настоящую улицу в `street`. Определить правило для New-Moscow без улицы.
- [ ] Юнит-тесты парсера на новые форматы (включая адреса без улицы).
- [ ] Команда/скрипт точечной переразборки битых записей (dry-run по умолчанию). На 2026-06-16 их было 6: `#2153, #2154, #2160, #2162, #2171, #2176` — **перепроверить запросом**, id могли измениться.
- [ ] `php artisan addresses:merge-duplicates` — схлопнуть всплывшие дубли с «чистыми», проверить перепривязку заявок.
- [ ] `php artisan addresses:geocode-missing` для переразобранных (после включения «Стандартизации» у токена DaData).
- [ ] Локально на копии прод-БД → проверка → деплой → на проде выполнить чистку/merge в порядке.

## [ ] 3. Оборудование под бригадой: «живой» режим без лага (частота + догруз при загрузке)

**Контекст.** В задаче по складу сделано: оборудование (комплект H-* + машина) показывается рядом
с бригадой для ОТКРЫТЫХ заявок только за сегодня, данные кэшируются в `request_equipment` и
наполняются командой `wms:refresh-equipment` (`routes/console.php`, сейчас `->hourly()`), при закрытии —
заморозка. Минус: между выдачей инвентаря на складе и появлением под бригадой — лаг (до часа).
Критичен момент утренней выдачи.

**Что решить/сделать (обсуждали 2026-06-22):**
- [ ] Поднять частоту cron `wms:refresh-equipment` (кандидаты: каждые 10 мин — как «пол»; либо 30 мин если будет догруз при загрузке).
- [ ] Догруз при загрузке дневного вида: после рендера страницы JS асинхронно зовёт новый роут
      `POST /api/requests/refresh-equipment-today` → тот обновляет `request_equipment` за сегодня и
      возвращает свежие данные, ячейки «Бригада» обновляются. Страница не блокируется на время опроса склада.
- [ ] Троттлинг на стороне роута: если обновляли < ~60 сек назад — отдать кэш, склад не дёргать
      (через `Cache`-метку времени последнего обновления). Иначе F5 будет долбить склад.
- [ ] Не ронять страницу, если склад недоступен (best-effort, try/catch).
- [ ] (Опц.) Альтернатива «по-настоящему живого»: батч-эндпоинт на складе (stock-flow),
      принимающий список email и отдающий оборудование разом — 1 HTTP-вызов на загрузку вместо N.
- [ ] **Прод-инфра (нужно для ЛЮБОГО варианта):** на сервере должен работать планировщик Laravel —
      `* * * * * cd /var/www/lan-install && php artisan schedule:run` в crontab. Сейчас не найден.
